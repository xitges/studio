## Plugin Hosting

### 스캔

플러그인 매니저 싱글턴이 백그라운드 스레드에서 VST3 / AU 검색 경로를 탐색. 탐색 결과는 메모리에 보관하고 앱 설정 저장소에 캐시. 앱 시작 시 캐시를 먼저 로드하고, 플러그인 디렉터리의 변경이 감지되면 자동 재스캔. 스캔 진행률과 완료 콜백은 메시지 스레드에서 발화해 UI를 안전하게 갱신할 수 있도록 한다.

### 로드 / 언로드

플러그인 인스턴스 생성은 메시지 스레드에서 수행한다. 생성이 완료된 뒤에야 락을 잡고 채널 슬롯에 교체한다. 기존 인스턴스는 락 밖에서 해제해 오디오 스레드의 락 대기 시간을 최소화한다.

```
로드 순서:
  1. 메시지 스레드에서 인스턴스 생성 + prepareToPlay
  2. CriticalSection lock 획득
  3. 채널 슬롯에 새 인스턴스 교체
  4. lock 해제 후 구 인스턴스 releaseResources
```

### 오디오 스레드 처리

```
오디오 콜백 per-channel
  → try-lock 시도
      성공: 플러그인에 MIDI 버퍼 + 오디오 버퍼 처리 위임
      실패: 해당 채널 무음 통과
```
NoteEvent를 MIDI 버퍼로 변환할 때 noteOff를 별도로 추적. 스텝이 끊기거나 패턴이 교체될 때 hanging note가 발생하지 않도록 하기 위해서.

### 상태 저장

저장 시 플러그인 자체 상태를 바이너리로 추출해 base64로 인코딩한 뒤 프로젝트 파일에 포함. 복원 시 역순으로 플러그인에 주입해 프리셋, 파라미터, 내부 상태를 재현.

## 4. Rendering / Export

실시간 오디오 파이프라인을 오프라인으로 구동해 동일한 결과를 WAV 파일로 내보낼 수 있도록 설계.

### 오프라인 렌더 원칙

별도 렌더 전용 경로를 두지 않는다. 실시간 재생에 사용하는 패턴 처리 → FX 체인 → 마스터 처리 파이프라인을 그대로 오프라인으로 반복 구동한다. 렌더 결과가 실시간 재생과 항상 동일하게 유지된다.

### 렌더 흐름

```
1. 실시간 오디오 콜백 분리          — 오디오 장치와의 연결 해제
2. 모든 voice / synth / FX 상태 초기화
3. WAV writer 생성 (24bit, 스테레오)
4. 오프라인 루프 (512샘플 블록 단위 반복)
     버퍼 초기화
     → 패턴 또는 송 파이프라인 처리
     → WAV 파일에 기록
5. 실시간 오디오 콜백 재연결
```

렌더 전에 실시간 콜백을 분리하는 이유는, 오디오 장치와 렌더러가 동시에 같은 엔진 상태를 구동하면 voice, 시퀀서 위치, FX 상태가 충돌하기 때문.

### 렌더 범위 계산

- Pattern Mode: 활성 패턴의 스텝 수를 bar로 환산
- Song Mode: 플레이리스트 클립 전체의 끝 위치를 기준

### Stem Export

트랙별로 렌더를 반복 호출해 트랙당 WAV 파일을 분리 생성. 각 호출 시 해당 트랙만 출력에 포함되도록 믹서 상태를 임시로 조정.

### 포맷

WAV, 24bit, 프로젝트 샘플레이트(기본 44100Hz), 스테레오.
